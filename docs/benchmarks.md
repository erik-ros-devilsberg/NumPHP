## How to read this

- **ratio** is `numphp_median / numpy_median`. Values **>1** mean
  numphp is slower; **<1** means faster; **~1** means parity.
- The matmul scenarios should be near-parity since both engines call
  the same OpenBLAS routine. Significant divergence there points at
  layout / copy overhead.
- Element-wise scenarios where NumPy is faster reflect NumPy's more
  vectorised inner loops; numphp uses a generic nd-iterator.
- `fromArray` / `toArray` are interop costs — building a numphp
  array from a nested PHP array (or going back) is unavoidable
  bridging work that fast paths can't hide.
- `slice (view creation)` should be sub-millisecond on both sides
  — both engines return zero-copy views, the timing measures pointer
  arithmetic + metadata setup.

## Reproduce

```bash
cd /home/arciitek/git/numphp
phpize && ./configure --enable-numphp
make CFLAGS="-Wall -Wextra -O2 -g"
bash bench/run.sh
```

The first `bench/run.sh` invocation creates `bench/.venv` and
`pip install`s NumPy into it (gitignored). Subsequent runs reuse
the venv.
